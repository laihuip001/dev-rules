# 🔄 事業運営（Flow開発）引き継ぎプロトコル

## 1. 目的

プロジェクトを「ルール整備フェーズ」から「実開発フェーズ」へ移行する際の、役割別アプローチを定義する。
久しぶりに開発を再開する際の**スタートアップ・ガイド**として使用する。

## 2. 役割別アクション

### 🏛️ 設計係 (Architect) のムーブ

**目的**: 現場が迷わない「完璧な設計図」を作ること。

1. **環境同期 (Sync)**:
    - ターミナルで `git pull` を実行。（最新のルールブックを入手）
2. **状況把握 (SITREP)**:
    - `task.md` を読み、次に倒すべき敵（タスク）を確認する。
3. **作戦立案 (Planning)**:
    - `implementation_plan.md` を新規作成/更新する。
    - **思考のポイント**:
        - 「どのファイルをどう変えるか？」
        - 「テストはどうするか？」
        - 「既存機能への影響は？」
4. **指示出し (Dispatch)**:
    - 計画書ができたら `git push`。「現場監督、あとは頼んだ」と念じる。

### 👷 現場監督 (Constructor) のムーブ

**目的**: 設計図通りに、堅牢なコードを組み上げること。

1. **環境同期 (Sync)**:
    - ターミナルで `git pull` を実行。（設計図が降りてくる）
2. **設計確認 (Review)**:
    - `implementation_plan.md` を読む。
    - 疑問があれば設計係（過去の自分、またはAI）に問いかける姿勢を持つ。
3. **実装 (Code)**:
    - `src/` 以下のコードを書く。
    - **TEALS**: ログが正しく記録されているか確認する。
4. **納品 (Deliver)**:
    - テストが通ったらコミット。「feat: 機能実装」
    - `task.md` のチェックボックスを埋めて `git push`。

## 3. 次回のFirst Action

PCを開いたら、まず以下の呪文を唱える（入力する）。

```bash
cd dev-rules
git pull
```

これで、あなたの **「自動人事システム」** は再び起動する。
